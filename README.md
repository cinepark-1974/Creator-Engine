## v2.5.3 핫픽스 (2026-05-12) — long requests 정책 대응 + 단계별 JSON 저장

### 진단 — 운영 중 발견된 2가지 결함

**결함 1. 캐릭터 바이블 생성 실패 — long requests 정책**

Anthropic API의 정책 적용으로 비스트리밍 호출이 차단되었다. 에러 메시지:

> "Streaming is required for operations that may take longer than 10 minutes."

원인 — v2.5.1에서 캐릭터 바이블 안정성을 위해 max_tokens를 16,000 → 24,000으로 상향했을 때, 이 변경이 Anthropic API의 long requests 정책 임계값을 넘었다. v2.5.1·v2.5.2 적용 후 운영 중 발견된 회귀 결함.

**결함 2. JSON 저장이 Scene 단계 이후에만 존재**

v2.3.5에서 JSON 저장 기능 도입 시 Scene/Treatment/완성 시점에만 버튼이 배치되어, **Core/Bible 단계에서 다음 단계 에러 발생 시 작업 손실 위험.** Mr. MOON이 실제 운영 중 Core 완료 후 Character Bible 단계에서 long requests 에러를 만나 작업 손실 직전 상황 발생.

### 축 1: 캐릭터 바이블 스트리밍 전환

`call_character_bible_single()` 함수의 4개 API 호출을 모두 스트리밍 방식으로 전환.

| 호출 위치 | 변경 전 | 변경 후 |
|---|---|---|
| 1차 시도 | `client.messages.create()` | `client.messages.stream()` + `get_final_message()` |
| max_tokens 잘림 재시도 | `client.messages.create()` | `client.messages.stream()` + `get_final_message()` |
| JSON 1차 재시도 (temp 0.15) | `client.messages.create()` | `client.messages.stream()` + `get_final_message()` |
| JSON 2차 재시도 (temp 0.05) | `client.messages.create()` | `client.messages.stream()` + `get_final_message()` |

스트리밍 호출은 API 응답을 토큰 단위로 받으므로 long requests 정책을 통과한다. 출력 스키마·max_tokens·temperature·재시도 로직 모두 v2.5.1·v2.5.2 그대로 유지.

### 축 2: 단계별 JSON 저장 버튼 4종 신설

| 신설 위치 | 시점 | 비고 |
|---|---|---|
| Brainstorm 완료 | Gate A 통과 / 미통과 양쪽 | 미통과 시에도 저장 가능 |
| Core Build 완료 | Gate B+C 통과 / 보류 양쪽 | 보류 상태에도 저장 가능 |
| Character Bible 완료 | 7명 생성 완료 후 | 가장 시급한 보호 시점 |
| Structure Build 완료 | Gate D 통과 / 미통과 양쪽 | 미통과 시에도 저장 가능 |

기존 Scene/Treatment/완성 시점 3개와 합쳐 **총 7개 시점**에서 JSON 저장 가능. 어느 단계에서 멈춰도 그 시점까지의 작업이 보존된다.

### 축 3: v2.3.5 하드코딩 정리

기존 3곳의 `save_project_to_json(project, engine_version="v2.3.5")` 호출을 디폴트 사용으로 변경. v2.3.5 마이그레이션 잔재 제거.

### 하위 호환성

- 캐릭터 바이블 출력 스키마 그대로 (스트리밍은 호출 방식만 변경)
- JSON 저장 양식 동일 (`save_project_to_json` 함수 무수정)
- main.py 시그니처 무수정
- 기존 v2.5.0/v2.5.1/v2.5.2 프로젝트 100% 호환

### 다음 사이클 의제

- JSON 저장 자동화 (각 단계 완료 시 자동으로 백업 파일 다운로드 트리거)

## v2.5.2 핫픽스 (2026-05-04) — Idea→Creator 인계 LOCKED 추출 확장

### 진단 — 「오랜만에」 검증에서 발견된 결정적 결함

> "Idea Engine에서 아이디어를 제공했을 때 LOCKED를 해도 주요 포인트가 바뀌는데, 이를 방지할 수 있는 방법은 없나?" — Mr. MOON

「오랜만에(가제)」 작품 검증에서 **18개 핵심 모티프 중 11개(61%)가 Creator 출력에서 휘발**되는 현상이 발견되었다. Mr. MOON의 풀버전 아이디어와 Idea Engine 산출물(IdeaDiagnostic)을 Creator JSON과 정밀 비교한 결과, **결함의 원인은 LOCKED 시스템의 강도가 아니라 데이터 인계 구조**임이 명확해졌다.

| 모티프 | Mr. MOON 입력 | Idea Engine 인식 | Creator LOCKED 등록 |
|---|---|---|---|
| 버즈 통화연결음 | ✓ | ✓ ("Hook 점수 9/10 독창성 근거") | ✗ 누락 |
| 노래방·LP 금지 | ✓ | ✓ ("음악 사용 규약 LOCK"으로 명시) | ✗ 누락 |
| 건물 중정 구조 | ✓ | ✓ ("두 타임라인 연결핀 후보") | ✗ 누락 |
| 시드니/런던 외국행 | ✓ | (산출물에 등장) | ✗ 누락 |
| 옛 단체사진 / 노트 발견 | ✓ | ✓ ("Creator 결정 의제로 등록") | ✗ 누락 |
| 마지막 노래 한 곡 시퀀스 | ✓ | ✓ ("정서적 보상 회로 정점") | ✗ 누락 |

**Idea Engine은 일을 잘했다.** 핵심 모티프를 정확히 인식했고, 심지어 자체적으로 "음악 사용 규약 LOCK", "결말 형식 LOCK", "포맷 LOCK"이라는 메타 LOCK까지 산출했다.

**문제는 인계 단계였다.** main.py(line 3729~3771)의 LOCKED 자동 추출 로직이 Idea Engine 시드의 **7개 키만 추출**하고, Idea Engine이 추가로 산출한 **5개 영역(확정된 핵심 결정, 음악 사용 규약, 시각 모티프, 결말 형식, Creator 결정 의제)을 무시**한 채 진행되었다. 결과적으로 Creator Engine이 받은 LOCKED 11개 항목은 모두 작품 외형(로그라인/장르/주제/타겟/회차/참고작/위험요소)에 한정되었고, 작품 본질을 결정하는 디테일은 LOCKED에 등록되지 못한 채 Brainstorm 단계부터 휘발되었다.

v2.5.2는 이 인계 결함을 두 갈래로 메운다.

### 축 1: main.py LOCKED 추출 영역 5종 신설

기존 7개 키 추출에 더해, Idea Engine 시드의 다음 5개 영역을 추가 추출:

| 신규 키 | Idea Engine 출처 | LOCKED 표시 형식 |
|---|---|---|
| `locked_core_decisions` | "확정된 핵심 결정" 블록 | `[★ 확정된 핵심 결정 LOCK ★]` |
| `locked_music_rules` | "음악 사용 규약 LOCK" | `[★ 음악 사용 규약 LOCK ★]` |
| `locked_visual_motifs` | Hook 보강 제안 + Creator Questions | `[★ 시각 모티프 LOCK ★]` |
| `locked_ending_form` | "결말 형식 LOCK" | `[★ 결말 형식 LOCK ★]` |
| `locked_creator_questions` | "Creator Engine에서 결정해야 할 질문" | `[Creator 결정 의제]` |

**fallback 메커니즘** — 신규 5개 키가 없는 구버전 Idea Engine 시드 JSON에서도 정상 작동. 신규 키가 부분적으로만 있어도 부분 추출 동작.

**보너스** — 기존 추출에서 누락되었던 `genre.tertiary` / `target.global` 키도 함께 추출하도록 보완.

### 축 2: LOCKED_SYSTEM_RULES에 4종 신규 보호 카테고리 명시

prompt.py의 LOCKED_SYSTEM_RULES에 v2.5.2 신규 4종 카테고리에 대한 절대 보존 룰을 명시. Brainstorm/Core/Character/Structure/Scene/Treatment 모든 단계가 이 표식을 작품 본질로 인식하고 보존하도록 강제.

```
1. [★ 확정된 핵심 결정 LOCK ★] — 작가가 작품 본질로 결정한 사항
   "더 흥행할 것 같은 대안을 제시하지 마라. 이미 결정된 사항이다."

2. [★ 음악 사용 규약 LOCK ★] — 음악·노래·OST 사용 방식
   "금지된 사용 방식을 어떤 변주로도 도입하지 마라."

3. [★ 시각 모티프 LOCK ★] — 두 타임라인·두 세계 연결 오브제
   "LOCK된 모티프는 반드시 작품에 등장시켜라."

4. [★ 결말 형식 LOCK ★] — 결말의 본질
   "절제 미학을 핑계로 결말 형식을 약화시키지 마라."
```

추가로 **LOCKED 검증 체크리스트에 5개 신규 항목** 추가 — 모든 단계가 출력 직전 4종 보호 카테고리 + Creator 결정 의제 답변 여부를 자가 점검.

### 검증 방법

다음 측정 지점을 통해 v2.5.2의 효과를 확인할 수 있다.

## v2.5.1 핫픽스 (2026-05-04) — 캐릭터 바이블 JSON 출력 안정화

### 진단 — Mr. MOON 지적

> "캐릭터 바이블 만들 때 양이 많다 보니까 한 번에 성공하지 못하고 두~세 번 시도를 하게 된다."

v2.5.0에서 `active_desire_actions` / `independent_desire` / `active_beat` / `active_beat_description` 4개 필드가 추가되어 캐릭터 바이블 페이로드가 무거워졌다. 결과적으로 다음 3가지 실패가 빈발했다:

1. **JSON 파싱 실패** — 대사 안 쌍따옴표·줄바꿈·역슬래시로 JSON 깨짐
2. **max_tokens 잘림** — 16,000 토큰 한계 초과
3. **1차 자동 재시도 후에도 실패 시 사용자 수동 재실행** — 작가 작업 흐름 단절

코드 점검 결과, 캐릭터 바이블은 이미 1인 단위로 순차 생성되고 있었고(v2.3.1 일관성 보호 구조), 한꺼번에 다수 캐릭터를 만드는 구조가 아니었다. 따라서 병목은 "동시 다발 생성"이 아닌 **개별 캐릭터 1인의 JSON 출력 안정성**이었다.

v2.5.1은 이 병목을 3축으로 메운다. 작업량 작고 무위험·고효과의 핫픽스 사이클.

### 축 1: CHAR_BIBLE_RULES에 9개 강제 JSON 출력 룰 신설

기존 룰셋은 "JSON으로 출력하라" 수준의 약한 지시였다. v2.5.1은 BAD/GOOD 예시와 자가 검증 5항목을 포함한 강제 룰로 강화.

**신설된 9개 룰**

| # | 룰 | 효과 |
|---|---|---|
| 1 | 단일 JSON 객체 (설명·주석·마크다운 펜스 금지) | 코드 펜스 출력 차단 |
| 2 | 모든 key는 쌍따옴표 | key 누락 차단 |
| 3 | 모든 string value는 쌍따옴표 | value 누락 차단 |
| 4 | string value 안의 인용은 작은따옴표만 | **가장 빈번한 실패 차단** |
| 5 | string value 안에 줄바꿈 금지 | JSON 깨짐 차단 |
| 6 | string value 안에 역슬래시 금지 | 이스케이프 깨짐 차단 |
| 7 | 후행 쉼표(trailing comma) 금지 | 파싱 실패 차단 |
| 8 | ```json 펜스로 감싸지 말 것 | safe_json_loads 호환 |
| 9 | 출력 시작 `{` / 출력 끝 `}` | 전후 텍스트 차단 |

**BAD/GOOD 예시 명문화**

```
❌ "sample_lines": {"normal": "그가 "안녕"이라고 말했다."}  ← JSON 깨짐
✅ "sample_lines": {"normal": "그가 '안녕'이라고 말했다."}  ← 정상

❌ "backstory": "어린 시절 가족을 잃었다.\n그 후 혼자 자랐다."  ← JSON 깨짐
✅ "backstory": "어린 시절 가족을 잃었다. 그 후 혼자 자랐다."  ← 정상
```

**자가 검증 5항목** — 출력 직전 모델이 스스로 점검하는 체크리스트. 1건이라도 위반되면 출력 전 수정 강제.

### 축 2: max_tokens 16,000 → 24,000

v2.5.0 신규 4개 필드로 응답 토큰이 평균 2,000~3,000 증가했다. 16,000 한계로는 여유 마진이 부족했음. 24,000으로 상향하여:

- 평균 응답에 충분한 마진 확보
- max_tokens 잘림 자동 재시도(분량 축소 모드) 진입 빈도 감소
- 실제 비용은 평균 사용량 변화 없음 (max_tokens는 한계값일 뿐 평균값이 아님)

### 축 3: JSON 파싱 자동 재시도 1회 → 2회 (temperature 단계적 하강)

기존: 1차 시도(temp 0.3) → 실패 시 1회 재시도(temp 0.2) → 그래도 실패하면 사용자에게 수동 재실행 요청.

v2.5.1: 1차 시도(temp 0.3) → **1차 재시도(temp 0.15)** → **2차 재시도(temp 0.05 + JSON_OUTPUT_RULES_STRICT 추가 주입)** → 그래도 실패하면 raw 디버그 표시.

**Temperature 단계적 하강의 효과**

| 단계 | temperature | 강조 메시지 |
|---|---|---|
| 1차 시도 | 0.3 | 기본 프롬프트 |
| 1차 재시도 | 0.15 | "★ 작은따옴표만 / 줄바꿈 금지 / 역슬래시 금지 / 후행 쉼표 금지 / 코드 펜스 금지 ★" |
| 2차 재시도 | 0.05 | JSON_OUTPUT_RULES_STRICT 전체 주입 + "절대 위반 금지" 명문화 |

낮은 temperature는 모델이 형식 룰을 더 엄격히 따르게 한다. 0.05까지 내려가면 사실상 결정적(deterministic) 출력에 가까워짐.

### 검증 방법

## v2.5.0 업데이트 (2026-05-04) — Writer Engine v3.5.1 환류 반영 + 5축 데이터 인터페이스

### 진단 — Writer Engine으로부터의 환류

> "Rewrite가 짚은 8개 결함 중 5개가 Writer Engine 단계가 아닌 Creator Engine 입력 단계의 미비에서 시작된 것으로 판단됨. Creator JSON 입력 자체가 부실하면 Writer 룰셋이 잡지 못하는 결함이 그대로 시나리오에 재현됨."

「오랜만에(가제)」 작품을 Writer Engine v3.3.0으로 집필한 결과 Rewrite Engine 분석에서 **7.1점/10.0(고려 등급)** 측정. Writer Engine은 v3.5.1로 자체 보강(A19 시간 표기 금지, A25 캐릭터 보이스 차별화, A26 중반 침식 차단, A27 POV 회전, OBSTACLE INTENSITY) 완료. 그러나 Writer 단에서 결함을 후처리하는 것은 한계가 있고, **Creator Engine이 출력하는 JSON 구조 자체가 Writer가 흡수할 수 있는 데이터 인터페이스를 갖춰야** 환류 루프가 닫힌다는 결론.

Writer Engine v3.5.1 코드를 점검한 결과 결정적 단서가 발견됨 — `build_scene_plan_prompt()` 함수의 시그니처에 `opening_strategy: str = ""  # ★ v3.6 신규 — Creator Engine 산출물` 라인이 이미 선반영되어 있음. **Writer는 이미 Creator로부터 데이터를 받을 인터페이스를 만들어두고 기다리고 있는 상태.** Creator Engine v2.5.0은 이 인터페이스를 채워주는 사이클.

v2.5.0은 이 공백을 **5축 데이터 인터페이스**로 메운다. 모든 장르에 적용된다 (멜로/로맨스 한정 아님).

### 축 1: OBSTACLE DESIGN — 장벽 시퀀스 사전 설계

이야기는 갈등(Conflict)만이 아니라 장벽(Obstacle)으로 작동한다. 갈등이 "두 의지의 충돌"이라면, 장벽은 "의지가 도달하지 못하게 막는 것"이다. 장벽이 비트 단위로 설계되지 않으면 인물들이 너무 자주/쉽게 만나·대결해·도달해서 긴장 곡선이 평탄해진다.

Rewrite 진단의 "수진이 매일 서점을 방문하는 구조여서 물리적 거리가 없고 장벽이 실제로 작동하는 순간은 S#66~67뿐이다"가 정확히 이 문제. v2.5는 Structure Build 단계에서 장벽을 비트별로 사전 설계한다.

**장벽 4유형**

| 유형 | 정의 | 장르별 작동 예 |
|---|---|---|
| physical | 물리적 거리·격리·봉쇄·체포·이동 불가 | 멜로의 지방 출장, 스릴러의 추적자 막힘 |
| psychological | 자기 제약·트라우마·죄책감·자존심 | 멜로의 마음 문 닫음, 드라마의 화해 거부 |
| status | 사회적 지위·정체 은폐·금지된 관계·계급 | 멜로의 신분 차이, 스릴러의 위장 신분 |
| temporal | 어긋남·엇갈림·데드라인·과거 봉인·시차 | 멜로의 한 발 늦음, 미스터리의 과거 봉인 |

**장벽 작동 원칙 (전 장르)**

| 원칙 | 강제 사항 |
|---|---|
| 장벽 비중 20~30% | 메인 인물이 목표에 도달하지 못하는 비트가 전체의 20~30% |
| Beat 6/10/13 에스컬레이션 | 1막→2막 / Midpoint 직후 / 3막 진입 직전 — 각 단계에서 장벽이 질적 강화 |
| 장벽 유형 다양화 | 한 작품에 최소 2개 이상의 장벽 유형을 교차 |
| 인물 능동성 자극 | 장벽이 새 행동을 촉발해야 함 (장벽 = 행동 다양성 엔진) |

**Structure Build 출력 의무 — `obstacles` 블록**

```json
"obstacles": {
  "barrier_types": ["physical", "status"],
  "barrier_beats": [3, 6, 10, 13],
  "barrier_escalation": {
    "beat_6": "첫 장벽 가시화 — 구체 묘사",
    "beat_10": "장벽 질적 강화 — 구체 묘사",
    "beat_13": "장벽 최고조 — 구체 묘사"
  },
  "barrier_action_response": "각 장벽에 대해 인물이 시도하는 새 행동"
}
```

이 필드는 Writer Engine v3.5.1의 OBSTACLE INTENSITY 모듈이 직접 흡수해 만남 빈도 50% 상한과 Beat 6/10/13 에스컬레이션을 자동 충족시킨다.

### 축 2: CHARACTER ACTIVE DESIRE — 양방향 능동 갈망 강제

캐릭터의 매력은 "수동적 반응"이 아니라 "능동적 추구"에서 온다. Rewrite 진단의 "재원이 '손님이시니까' '커피 드릴까요'를 반복할 뿐 능동적 선택을 거의 하지 않아 멜로 상대역으로서의 매력이 축적되지 않는다"가 정확히 이 문제.

기존 Creator Engine은 주인공의 능동 동기는 잘 잡지만, 상대역(antagonist/ally/mirror, 특히 멜로 카운터파트)이 "주인공이 시키는 일에 반응만 하는 인물"로 전락하는 경향이 있었다. v2.5는 모든 메인 캐릭터에 능동 행동 3개 이상을 강제 명시한다.

**Character Bible 출력 의무 — `active_desire_actions` 필드**

모든 메인 역할(protagonist / antagonist / ally / mirror)에 강제:

```json
"active_desire_actions": [
  "이 인물이 자발적으로 하는 행동 1 — 동기 명시",
  "이 인물이 자발적으로 하는 행동 2 — 동기 명시",
  "이 인물이 자발적으로 하는 행동 3 — 동기 명시"
]
```

**작성 조건 3가지**

| 조건 | 의미 |
|---|---|
| 능동성 | 주인공이 시켜서가 아니라 본인이 자발적으로 하는 행동 |
| 구체성 | "사랑한다", "싫어한다" 같은 추상이 아닌 구체적 동작 |
| 정체성 | 그 행동이 그 인물만의 방식이어야 함 (다른 인물로 대체 불가) |

이 필드는 Writer Engine v3.5.1의 A27(POV ROTATION) 룰셋과 직접 연동된다. A27은 "각 메인 인물의 단독 씬을 최소 2씬 이상 배치"를 강제하는데, 그 단독 씬에서 인물이 무엇을 능동적으로 할지 = 이 필드가 결정한다.

### 축 3: PLANT DISTRIBUTION — 핵심 단서 분산 배치

기존 Plant & Payoff 룰은 "어떤 단서를 심을지"는 잡지만, "언제 어디에 분산 배치할지"는 명시하지 않았다. 결과적으로 핵심 단서가 후반에 몰려서, 그 전까지 관객에게 감정적 근거가 부재한 상태.

Rewrite 진단의 "20년 전 관계의 정체가 S#52~55에서야 암시되고 S#67에서 직접 언급되는데, 그 전까지 관객에게 이 재회가 왜 특별한지 감정적 근거가 주어지지 않는다"가 정확히 이 문제.

v2.5는 모든 Plant 항목에 비트 번호를 강제하고, "관객이 추론할 수 있는 시점(audience_inference_beat)"을 별도 명시한다.

**Core Build 출력 의무 — `planting_payoff` 스키마 확장**

```json
"planting_payoff": [
  {
    "type": "character | relationship | world",
    "plant_description": "심어지는 단서 내용",
    "plant_beat": 2,
    "audience_inference_beat": 9,
    "payoff_beat": 13,
    "payoff_description": "회수 시 의미"
  }
]
```

**분산 배치 강제**

| 권장 분산 | 의미 |
|---|---|
| Plant 1: Beat 1~3 | 1막 초반 (캐릭터 Plant) |
| Plant 2: Beat 3~6 | 1막 중후반 (관계 Plant) |
| Plant 3: Beat 6~9 | 2막 초중반 (세계관 Plant) |

3개 Plant가 모두 같은 비트(±1)에 몰리면 Structure Diagnosis 단계에서 "PLANT 분산 실패" 경고. `audience_inference_beat`이 `plant_beat`과 같으면 "PLANT 노골화", `payoff_beat`과 같으면 "PLANT 추론 시점 부재" 경고.

### 축 4: EMOTIONAL CLIMAX DESIGN — 감정 폭발 지점 사전 설계

Beat Sheet에서 "감정 폭발 비트"가 명시되지 않으면, 한국 영화 특유의 절제·침묵 미학이 과잉 적용되어 폭발 지점이 흐릿해진다. Rewrite 진단의 "감정이 쌓이기는 하나 명확한 폭발 지점이 없다. S#75 고백 장면도 재원의 감정적 반응이 절제를 넘어 부재에 가까워 카타르시스가 약하다"가 정확히 이 문제.

v2.5는 Structure Build 단계에서 폭발 비트 + 폭발 형식 + 감정 온도 변화를 사전 결정한다.

**감정 폭발 7형식 (전 장르)**

| 형식 | 주 적용 장르 |
|---|---|
| confession (고백) | 멜로 / 드라마 |
| confrontation (정면 대결) | 스릴러 / 액션 / 드라마 |
| reveal (진실 폭로) | 미스터리 / 스릴러 / 드라마 |
| parting (결별·이별) | 멜로 / 드라마 |
| breakthrough (자기 돌파·각성) | 모든 장르 |
| sacrifice (희생 결단) | 액션 / 드라마 / SF |
| reconciliation (화해·용서) | 드라마 / 멜로 / 가족 |

**Structure Build 출력 의무 — `emotional_climax` 블록**

```json
"emotional_climax": {
  "beat": 13,
  "form": "confession",
  "pre_temperature": 3,
  "post_temperature": 9,
  "temperature_delta": 6,
  "description": "폭발의 구체적 묘사 2~3문장",
  "genre_alignment": "이 폭발이 장르 약속과 어떻게 일치하는가"
}
```

**핵심 검증** — `temperature_delta`가 4 미만이면 실질적 폭발이 아닌 것으로 판정. Structure Build Gate에서 -1점 감점.

**절제형 엔딩에서도 폭발 1회 강제** — "한국적 절제 엔딩"을 선택해도 Beat 11~13 어딘가에서 폭발 1회는 반드시 일어나야 한다. 절제는 폭발 후의 잔향이지, 폭발 자체의 부재가 아니다.

### 축 5: B-STORY INDEPENDENCE — 서브 캐릭터 독립 동기

서브 캐릭터(supporting / catalyst / subplot_lead)가 "메인 플롯의 보조자·관찰자·공감자"로만 처리되면, 후반에 그 인물이 독립 행동을 하거나 감정을 드러낼 때 뜬금없이 느껴진다. Rewrite 진단의 "서하가 재원의 보조이자 관찰자로 기능하지만, S#81에서 '그분 말고요'라는 대사가 암시하는 서하 자신의 감정이 이전 시퀀스에서 충분히 축적되지 않았다"가 정확히 이 문제.

v2.5는 supporting 역할에 메인 플롯과 별개의 독립 욕망을 강제한다.

**Character Bible 출력 의무 — supporting role 추가 필드**

역할이 supporting / catalyst / subplot_lead / mentor / rival / informant 인 경우:

```json
"independent_desire": "메인 플롯과 별개로 이 인물이 추구하는 것 (1~2문장)",
"active_beat": 8,
"active_beat_description": "그 비트에서의 행동 + 메인 플롯에 미치는 영향"
```

**단순 조력자 패턴 명시적 차단**

| 차단 패턴 | 권장 패턴 |
|---|---|
| ❌ "주인공의 가장 친한 친구. 주인공을 응원한다." | ✅ "주인공의 친구. 자신의 이혼 소송 중 주인공의 사랑을 보며 갈등한다." |
| ❌ "주인공의 동료. 주인공을 돕는다." | ✅ "주인공의 동료. 자기 승진 누락에 대한 분노를 주인공 사건에 투영한다." |
| ❌ "주인공의 가족. 주인공을 걱정한다." | ✅ "주인공의 형. 동생의 성공이 자신의 실패를 증명할까 두렵다." |

이 필드는 Writer Engine v3.5.1의 A26(MIDPOINT EROSION) 룰셋과 연동된다. A26은 "중반(Beat 6~10)에 B스토리가 메인 플롯을 흔드는 사건 최소 1회"를 강제하는데, 그 사건의 동력 = `independent_desire`가 만든 `active_beat`.

## v2.3.7 업데이트 (2026-04-22) — 시대 고증 검증 추가

**C. Gate E 채점 축 7개 → 8개 확장**

- 호러: 2막 후반 물리적 공포 장면 최소 2개+, 유령 인식 범위 명시, 위협 반복 등장

### 효과 — Writer Engine 가동 조건 완성

- `CLIMAX_FAIL` = structure 항목 **-2점**

## BLUE JEANS NARRATIVE DRIVE (BJND) — 엔진의 척추

### BJND 용어 표준 (전 엔진 공통 어휘)

```
발생요인(Loss/Lack) → 욕망(Desire: Goal+Need) → 해결전략(Strategy) → 대가(Cost)
```

- **Loss(상실)** — 가졌다가 잃은 것. 욕망은 회복/복수/대체로 향함.
- **Lack(결핍)** — 처음부터 없던 것. 욕망은 획득/성장/증명으로 향함.
- **Desire(욕망)** — 외적 Goal(목적) + 내적 Need(필요)
- **Strategy(해결방법)** ★ — 이 인물만의 창의적 해결. **작가의 서명**이 남는 축.
- **Cost(대가)** — 이 전략이 치르게 하는 것. 2막 위기의 원천.

보조 필드: Risk / Ending Payoff / Goal↔Need Gap

### BJND의 끝까지 관통 원칙

BJND는 Core Build에서만 설계되지 않습니다. 전 파이프라인에 강제 전파됩니다.

- **Scene Design**: 각 씬에서 Strategy가 구체적 행동으로 드러남
- **Treatment**: 각 비트에서 Cost 누적 / Strategy 전환 강제
- **엔딩**: Strategy_1 → Strategy_2 전환이 반드시 구현

BJND가 중간에 사라지면 기획서는 강한데 시나리오는 약해집니다.

## v2.3 핵심 — 4축 자가 검증 엔진

엔진이 BJND를 설계한 후, 창작자 관점에서 4축으로 자기 진단합니다.

### ① NECESSITY (필연성)
> "이 인물이 왜 지금 이렇게 살아야만 했는가?"

Lack/Loss가 막연한 설정이면 실패. 구체적 사건·기억·행동으로 인물을 관통해야 합니다.

### ② AUTHENTICITY (인물 고유성) ★ 작가의 서명
> "이 Strategy는 이 인물이 아니면 불가능한가?"

**3중 검증**을 통과해야 합격:
- **치환 테스트**: 다른 작품 주인공에게 붙여도 성립하면 실패
- **유기성 테스트**: 직업 + Lack + Strategy가 한 문장으로 연결되어야 함
- **작가 서명 테스트**: "왜 이 인물은 이렇게 해결하는가"에 작가 시선이 담긴 정당화

### ③ EMPATHY (관객 공감력)
> "관객이 이 인물의 Cost를 보며 자기 자신을 떠올릴 것인가?"

동시대 관객의 보편 상처를 건드리지 못하면 남의 이야기가 됩니다.

### ④ POTENCY (변화 진폭)
> "이 인물의 변화가 관객 마음에 얼마나 큰 흔적을 남길 것인가?"

1막 Strategy_1 → 3막 Strategy_2의 변화가 내적이고 진폭이 커야 관객 마음에 남습니다.

### 종합 자가 진단 — 관객의 얼굴 그리기

4축 설계 후 엔진은 스스로 묻습니다:
1. 관객이 웃을 장면이 구체적으로 있는가?
2. 관객이 가슴 먹먹해질 장면이 구체적으로 있는가?
3. 관객이 '나였다면'을 고민할 장면이 있는가?
4. 영화 끝난 후 관객의 입에 남을 장면이 있는가?

답이 구체적으로 나오지 않으면 캐릭터 재설계.

## 창작자 감성 3요소 (v2.3 신규)

기술적으로 정확한 씬이 아니라, 관객의 몸과 마음이 반응하는 씬을 만드는 미학 원칙.

### PHYSICALITY OF EMPATHY — 공감의 물리성
모든 중요 씬은 관객의 신체 반응을 만들어야 합니다:  
호흡 멎음 / 가슴 뜨거워짐 / 눈 젖음 / 웃음 터짐 / 등골 서늘함 / 손바닥 땀

### SILENCE DESIGN — 침묵의 설계
가장 중요한 감정은 대사가 아닌 침묵으로 전합니다.
- **거부의 침묵** — 답할 수 있지만 답하지 않음
- **무력의 침묵** — 답하고 싶지만 답하지 못함
- **충만의 침묵** — 말이 필요 없는 이해

### PLANT AESTHETICS — 씨앗의 미학
1막의 작은 디테일이 3막에 폭발하는 관객의 쾌감.  
최소 3쌍: 캐릭터 Plant / 관계 Plant / 세계 Plant

## OPENING MASTERY (v2.2에서 유지)

첫 3분 안에 관객을 잡지 못하면 영화가 끝입니다.  
2020년 이후 한국 관객은 첫 3분 안에 영화를 판단합니다.

### 오프닝 6기법
- **Action Drop** — 이미 진행 중인 액션 안으로 떨어뜨리기
- **Cold Open** — 본편과 다른 시점/장면으로 시작
- **Tease & Reveal** — 수수께끼 이미지 후 회수
- **In Media Res** — 미래의 결정적 순간 먼저
- **Character Reveal Action** — 주인공을 정의하는 첫 행동
- **Hook Dialogue** — 강렬한 한 줄 대사

### 장르 = 오프닝 DNA

장르 본질이 오프닝의 DNA를 결정합니다.

**복합 장르 법칙**: 두 번째 장르(뒤에 붙은) = 본질
- 로맨틱 **코미디** → 코미디 DNA (첫 3분 웃음)
- 액션 **스릴러** → 스릴러 DNA (첫 3분 긴장)
- 코믹 **로맨스** → 로맨스 DNA (첫 3분 신체 언어)
- 시대극 **액션** → 액션 DNA (첫 3분 물리 쾌감)

### 오프닝 ≠ 도발적 사건

한국 상업영화 표준 3단 구조:
```
오프닝 (0~3분)         = 장르·캐릭터·톤의 선언
Set-Up (3~10분)        = 일상 속 결핍·욕망 축적  
Inciting Incident (10~15분) = 도발적 사건
```

첫 3분에 사건을 터뜨리면 Set-Up이 사라져 관객 공감이 생기지 않습니다.

## 9단계 파이프라인

```
① 아이디어 입력 + LOCKED 설정
② Brainstorm (컨셉 카드 10개 + BJND 다양성 + Gate A)      [Sonnet]
③ Core Build (BJND 4단 + 4축 자가검증 + 오프닝 전략)      [Sonnet]
④ Character Bible (백스토리·전술·말투·대사·변화궤적)      [Opus]
⑤ Structure Build (Synopsis + Storyline + 3막 진단)        [Sonnet]
⑥ Scene Design (핵심 장면 + BJND 씬 레벨 집행)             [Sonnet]
⑦ Treatment Build (16비트 다중씬 + Cost 누적 + 창작자 감성) [Opus]
⑧ Tone Document (카메라·색감·페이싱·대사 규칙)              [Opus]
⑨ Export (기획개발보고서 DOCX)
```

### 이중 모델 (Opus / Sonnet)

| 단계 | 모델 | 이유 |
|---|---|---|
| Brainstorm · Core · Structure · Scene · Gate | Sonnet | 구조 작업 — 비용 효율 |
| Character Bible · Treatment · Tone | **Opus** | 서사 품질 — 대사/묘사/감정 |

Opus 호출 시 JSON 파싱 실패 → 자동 재시도 1회 (temperature 낮춤 + JSON 규칙 강화)

## LOCKED 시스템

```
🔒 LOCKED — 파이프라인 전 과정에서 절대 변경 불가
   장르: 로맨틱 코미디 (스릴러/드라마로 전환 금지)
   주인공: 강유진 (32, 여)
   핵심 관계: 아버지의 상속 조건 — "진짜 사랑하는 남자를 데려와라"
   공간: 쿠킹 클래스 (아이반 + 성인반)
   테마: 사람은 요리처럼 섞을 수 없다
   엔딩 방향: 한 명 선택 아닌 자기 발견
```

**v2.3부터 OPEN 필드 폐지.** 과거 버전의 OPEN 필드는 "AI가 창작 가능한 영역"이라는 의미였지만, 실제로는 **LOCKED에 없는 모든 것은 이미 창작 가능**합니다. 이중 구조가 혼란만 주었기에 제거했습니다.

**원칙**: 반드시 지킬 것만 LOCKED에 명시하세요. 나머지는 엔진이 자유롭게 창작합니다.

- 프로젝트 생성 시 입력 + 프로젝트 뷰에서 편집 가능
- 모든 API 호출에 자동 주입
- 기존 프로젝트의 open_items는 무시되므로 하위 호환성 유지

## Planting & Payoff

```
③ Core Build    → planting_payoff 3쌍 설계 (character / relationship / world)
⑥ Scene Design  → 각 장면에 plant_payoff_tag
⑦ Treatment     → 각 비트에 plant_payoff 회수 체크

1막에 심은 것이 3막에서 새로운 의미로 회수되어야 합니다.
