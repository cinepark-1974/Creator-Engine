## 정체성 선언

> **"작가의 시선으로 설계하고, 관객의 심장으로 검증하는 엔진"**

생성만 하는 AI 엔진이 아닙니다. 자기 설계를 스스로 검증하고 재생성하는 창작 파트너입니다.

다른 AI 도구는 "평균적으로 좋은 것"을 만들지만, Creator Engine은 **이 작품만의 서명이 남는 것**을 만듭니다. 작가의 시선이 닿은 해결 방식(Strategy)이 없는 시나리오는 실패로 판정합니다.

## 3-Engine Pipeline

```
[1] Creator Engine  → 아이디어 → 기획개발 패키지   (v2.3)
```

Creator Engine의 출력물이 모든 하위 엔진의 입력입니다.  
Creator Engine이 9.0 이상이어야 Writer Engine도 의미가 있습니다.

## BLUE JEANS NARRATIVE DRIVE (BJND) — 엔진의 척추

### BJND 용어 표준 (전 엔진 공통 어휘)

```
발생요인(Loss/Lack) → 욕망(Desire: Goal+Need) → 해결전략(Strategy) → 대가(Cost)
```

- **Loss(상실)** — 가졌다가 잃은 것. 욕망은 회복/복수/대체로 향함.
- **Lack(결핍)** — 처음부터 없던 것. 욕망은 획득/성장/증명으로 향함.
- **Desire(욕망)** — 외적 Goal(목적) + 내적 Need(필요)
- **Strategy(해결방법)** ★ — 이 인물만의 창의적 해결. **작가의 서명**이 남는 축.
- **Cost(대가)** — 이 전략이 치르게 하는 것. 2막 위기의 원천.

보조 필드: Risk / Ending Payoff / Goal↔Need Gap

### BJND의 끝까지 관통 원칙

BJND는 Core Build에서만 설계되지 않습니다. 전 파이프라인에 강제 전파됩니다.

- **Scene Design**: 각 씬에서 Strategy가 구체적 행동으로 드러남
- **Treatment**: 각 비트에서 Cost 누적 / Strategy 전환 강제
- **엔딩**: Strategy_1 → Strategy_2 전환이 반드시 구현

BJND가 중간에 사라지면 기획서는 강한데 시나리오는 약해집니다.

## v2.3 핵심 — 4축 자가 검증 엔진

엔진이 BJND를 설계한 후, 창작자 관점에서 4축으로 자기 진단합니다.

### ① NECESSITY (필연성)
> "이 인물이 왜 지금 이렇게 살아야만 했는가?"

Lack/Loss가 막연한 설정이면 실패. 구체적 사건·기억·행동으로 인물을 관통해야 합니다.

### ② AUTHENTICITY (인물 고유성) ★ 작가의 서명
> "이 Strategy는 이 인물이 아니면 불가능한가?"

**3중 검증**을 통과해야 합격:
- **치환 테스트**: 다른 작품 주인공에게 붙여도 성립하면 실패
- **유기성 테스트**: 직업 + Lack + Strategy가 한 문장으로 연결되어야 함
- **작가 서명 테스트**: "왜 이 인물은 이렇게 해결하는가"에 작가 시선이 담긴 정당화

### ③ EMPATHY (관객 공감력)
> "관객이 이 인물의 Cost를 보며 자기 자신을 떠올릴 것인가?"

동시대 관객의 보편 상처를 건드리지 못하면 남의 이야기가 됩니다.

### ④ POTENCY (변화 진폭)
> "이 인물의 변화가 관객 마음에 얼마나 큰 흔적을 남길 것인가?"

1막 Strategy_1 → 3막 Strategy_2의 변화가 내적이고 진폭이 커야 관객 마음에 남습니다.

### 종합 자가 진단 — 관객의 얼굴 그리기

4축 설계 후 엔진은 스스로 묻습니다:
1. 관객이 웃을 장면이 구체적으로 있는가?
2. 관객이 가슴 먹먹해질 장면이 구체적으로 있는가?
3. 관객이 '나였다면'을 고민할 장면이 있는가?
4. 영화 끝난 후 관객의 입에 남을 장면이 있는가?

답이 구체적으로 나오지 않으면 캐릭터 재설계.

## 창작자 감성 3요소 (v2.3 신규)

기술적으로 정확한 씬이 아니라, 관객의 몸과 마음이 반응하는 씬을 만드는 미학 원칙.

### PHYSICALITY OF EMPATHY — 공감의 물리성
모든 중요 씬은 관객의 신체 반응을 만들어야 합니다:  
호흡 멎음 / 가슴 뜨거워짐 / 눈 젖음 / 웃음 터짐 / 등골 서늘함 / 손바닥 땀

### SILENCE DESIGN — 침묵의 설계
가장 중요한 감정은 대사가 아닌 침묵으로 전합니다.
- **거부의 침묵** — 답할 수 있지만 답하지 않음
- **무력의 침묵** — 답하고 싶지만 답하지 못함
- **충만의 침묵** — 말이 필요 없는 이해

### PLANT AESTHETICS — 씨앗의 미학
1막의 작은 디테일이 3막에 폭발하는 관객의 쾌감.  
최소 3쌍: 캐릭터 Plant / 관계 Plant / 세계 Plant

## OPENING MASTERY (v2.2에서 유지)

첫 3분 안에 관객을 잡지 못하면 영화가 끝입니다.  
2020년 이후 한국 관객은 첫 3분 안에 영화를 판단합니다.

### 오프닝 6기법
- **Action Drop** — 이미 진행 중인 액션 안으로 떨어뜨리기
- **Cold Open** — 본편과 다른 시점/장면으로 시작
- **Tease & Reveal** — 수수께끼 이미지 후 회수
- **In Media Res** — 미래의 결정적 순간 먼저
- **Character Reveal Action** — 주인공을 정의하는 첫 행동
- **Hook Dialogue** — 강렬한 한 줄 대사

### 장르 = 오프닝 DNA

장르 본질이 오프닝의 DNA를 결정합니다.

**복합 장르 법칙**: 두 번째 장르(뒤에 붙은) = 본질
- 로맨틱 **코미디** → 코미디 DNA (첫 3분 웃음)
- 액션 **스릴러** → 스릴러 DNA (첫 3분 긴장)
- 코믹 **로맨스** → 로맨스 DNA (첫 3분 신체 언어)
- 시대극 **액션** → 액션 DNA (첫 3분 물리 쾌감)

### 오프닝 ≠ 도발적 사건

한국 상업영화 표준 3단 구조:
```
오프닝 (0~3분)         = 장르·캐릭터·톤의 선언
Set-Up (3~10분)        = 일상 속 결핍·욕망 축적  
Inciting Incident (10~15분) = 도발적 사건
```

첫 3분에 사건을 터뜨리면 Set-Up이 사라져 관객 공감이 생기지 않습니다.

## 9단계 파이프라인

```
① 아이디어 입력 + LOCKED 설정
② Brainstorm (컨셉 카드 10개 + BJND 다양성 + Gate A)      [Sonnet]
③ Core Build (BJND 4단 + 4축 자가검증 + 오프닝 전략)      [Sonnet]
④ Character Bible (백스토리·전술·말투·대사·변화궤적)      [Opus]
⑤ Structure Build (Synopsis + Storyline + 3막 진단)        [Sonnet]
⑥ Scene Design (핵심 장면 + BJND 씬 레벨 집행)             [Sonnet]
⑦ Treatment Build (16비트 다중씬 + Cost 누적 + 창작자 감성) [Opus]
⑧ Tone Document (카메라·색감·페이싱·대사 규칙)              [Opus]
⑨ Export (기획개발보고서 DOCX)
```

### 이중 모델 (Opus / Sonnet)

| 단계 | 모델 | 이유 |
|---|---|---|
| Brainstorm · Core · Structure · Scene · Gate | Sonnet | 구조 작업 — 비용 효율 |
| Character Bible · Treatment · Tone | **Opus** | 서사 품질 — 대사/묘사/감정 |

Opus 호출 시 JSON 파싱 실패 → 자동 재시도 1회 (temperature 낮춤 + JSON 규칙 강화)

## LOCKED 시스템

```
🔒 LOCKED — 파이프라인 전 과정에서 절대 변경 불가
   장르: 로맨틱 코미디 (스릴러/드라마로 전환 금지)
   주인공: 강유진 (32, 여)
   핵심 관계: 아버지의 상속 조건 — "진짜 사랑하는 남자를 데려와라"
   공간: 쿠킹 클래스 (아이반 + 성인반)
   테마: 사람은 요리처럼 섞을 수 없다
   엔딩 방향: 한 명 선택 아닌 자기 발견
```

- 프로젝트 생성 시 입력 + 프로젝트 뷰에서 편집 가능
- 모든 API 호출에 자동 주입
- v2.3부터 OPEN 필드는 폐지 예정 (의미 없는 "열린 공간"보다 명확한 LOCKED가 효과적)

## Planting & Payoff

```
③ Core Build    → planting_payoff 3쌍 설계 (character / relationship / world)
⑥ Scene Design  → 각 장면에 plant_payoff_tag
⑦ Treatment     → 각 비트에 plant_payoff 회수 체크
```

1막에 심은 것이 3막에서 새로운 의미로 회수되어야 합니다.  
회수 없는 Plant = 불만족 / Plant 없는 Payoff = 금지.
